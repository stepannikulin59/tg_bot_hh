# Техническое задание: Telegram-бот мониторинга вакансий HeadHunter

**Объект:** программный комплекс на языке Python.  
**Нормативные источники по API и правилам клиента:** [OpenAPI hh.ru](https://api.hh.ru/openapi/redoc), [dev.hh.ru](https://dev.hh.ru/), [hhru/api](https://github.com/hhru/api).  
**Нормативные источники по Telegram:** [python-telegram-bot](https://docs.python-telegram-bot.org/), пакет с очередью задач `python-telegram-bot[job-queue]`.

---

## 1. Цель и границы

**Цель:** автоматизированная выдача в Telegram списка и уведомлений о вакансиях с hh.ru по заданным правилам фильтрации.

**Границы:** один пользователь бота; настройка ключевых слов и идентификаторов — из конфигурации приложения и переменных окружения, без отдельного сценария редактирования фильтров через чат в рамках данного ТЗ.

---

## 2. Термины и обозначения


| Термин              | Значение                                                                                                                                                                       |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Состояние (state)   | Персистентные данные: `chat_id`, до **1000** `vacancy_id` (FIFO, FR-04, FR-04b), пороги **FR-04c** для пагинации |
| Подходящая вакансия | Вакансия, удовлетворяющая всем требованиям раздела 4                                                                                                                           |

## 3. Требования к интеграции с hh.ru

**INT-01.** Все HTTP-запросы к `api.hh.ru` выполняются с заголовком клиента в формате, **требуемом актуальной версией OpenAPI** (часто это имя приложения и контакт разработчика; в спецификации может встречаться заголовок **`HH-User-Agent`** или требование к **`User-Agent`** — не смешивать произвольно). Значение задаётся через переменную окружения (например `HH_USER_AGENT`), строка передаётся в тот заголовок(и), который указан в OpenAPI для вызываемого метода.

**INT-02.** Поиск вакансий: метод и параметры — по спецификации OpenAPI для `GET /vacancies`.

**INT-03.** Получение полной карточки при необходимости: `GET /vacancies/{vacancy_id}` по OpenAPI.

**INT-04.** Идентификатор области «Пермь» определяется по справочнику `GET /areas` либо задаётся в конфигурации (`PERM_AREA_ID`) после однократной сверки со справочником.

**INT-05.** Код расписания **удалённая работа** для ветки FR-15 берётся из справочника `schedule` в `GET https://api.hh.ru/dictionaries` (или эквивалент в OpenAPI); допускается фиксация в конфиге (`SCHEDULE_REMOTE_ID`) после однократной сверки со справочником.

**INT-06 (пагинация hh.ru).** У поисковых выдач с параметрами `page` и `per_page` действует ограничение глубины: суммарно доступно не более **2000** позиций в индексации (см. раздел про пагинацию в [OpenAPI](https://api.hh.ru/openapi/redoc)); запросы за пределом лимита возвращают ошибку — обход страниц **не** продолжать, зафиксировать в логах.

## 4. Функциональные требования

### 4.0. Логика работы сервиса

**L-01.** Данные о вакансиях для отбора берутся **только** с `api.hh.ru` (запросы по INT-02–INT-05 и полные карточки по INT-03 при необходимости). Локальное состояние **не** подменяет собой выдачу API при определении «какие сейчас самые свежие на hh.ru».

**L-02. Команда `/start` (каждый раз одинаково):** получить актуальный набор **подходящих** вакансий (подразделы 4.3–4.5) с объединением и сортировкой по **FR-15**; взять **не более 10** первых записей после сортировки по **`published_at` по убыванию**; отправить их в чат. Хранилище FR-04 **не участвует** в сортировке и отборе этих 10 (допускается, что все 10 уже есть в FR-04 — всё равно показываются снова). Если подходящих меньше 10 — отправляется фактическое число.

**L-03. Периодический опрос (каждые 300 с):** получить подходящие вакансии тем же правилом фильтрации (**включая постраночный обход** выдачи `GET /vacancies` по **обоим** базовым запросам FR-15 с **ранней остановкой** **FR-15b** и лимитом **NFR-05a**); для каждой найденной подходящей вакансии, чей `vacancy_id` **ещё не** содержится в хранилище FR-04, отправить одно автоуведомление (порядок отправки за тик — по **`published_at` по убыванию**, от более свежих к более старым), **добавить** id в FR-04 и выполнить **FR-04b**. Вакансии, чей id в текущем составе FR-04, автоопросом повторно не отправляются.

### 4.1. Общие

**FR-01.** Бот обрабатывает команду `/start` от пользователя Telegram.

**FR-02.** Идентификатор чата (`chat_id`) не задаётся в `.env`. При **первом** успешном `/start` значение `update.effective_chat.id` сохраняется в state. Повторные `/start` с **того же** чата обновляют только поведение выдачи (**L-02**). Запрос из **другого** `chat_id` (бот рассчитан на одного пользователя): **не** менять сохранённый `chat_id`, ответить кратким сообщением, что бот персональный.

**FR-03.** Токен бота Telegram хранится в `.env` (единое имя переменной — по соглашению в репозитории, например `TELEGRAM_BOT_TOKEN` или `BOT_TOKEN`; дублировать в `.env.example` без секрета).

**FR-04.** Персистентное состояние: поле `chat_id` (nullable до первого `/start`) и хранилище уникальных строковых `vacancy_id`, по которым уже сработала логика «не слать повторно с автоопроса» (см. FR-06). Хранится **не более 1000** id; поддерживается **порядок первого добавления** (FIFO) для политики вытеснения **FR-04b**.

**FR-04a.** Хранилище FR-04 **не используется** для ограничения состава выдачи по `/start` — см. **L-02**.

**FR-04b.** Если после добавления id (по **FR-06** или **FR-11**) число уникальных записей в FR-04 превышает **1000**, удалять из хранилища **самые ранние по порядку добавления** записи, пока не останется ровно **1000**. Id, вытесненные таким образом, **могут снова** попасть в чат через **автоопрос** при следующем обнаружении (NFR-08).

**FR-04c.** Для ранней остановки пагинации (**FR-15b**) в state дополнительно хранятся два nullable-поля (ISO 8601, как в ответе API): **`pagination_floor_perm`** и **`pagination_floor_remote`** — «пол» минимального `published_at` среди вакансий, **уже загруженных с API** по соответствующей ветке в **предыдущем завершённом** цикле **L-03**. До первого такого цикла значения **отсутствуют** (`null`). Поля обновляются в конце цикла **L-03**; при обработке **`/start`** поля **читаются** для **FR-15b**, но **не обновляются** (чтобы порог отражал именно глубину **опроса**). При **выбросе** id из FR-04 по **FR-04b** рекомендуется **сбросить** оба `pagination_floor_*` в `null` (или выполнить полный проход без ранней остановки по дате до следующего стабильного цикла), иначе «пол» может рассинхронизироваться с фактическим FR-04.

**FR-05.** Периодический опрос API hh.ru выполняется с интервалом **300 секунд** (5 минут). Опрос не должен приводить к отправке сообщений, если `chat_id` в состоянии отсутствует (до первого `/start`), либо опрос планируется только после первого успешного `/start` — на выбор реализации при условии соблюдения FR-06.

**FR-06.** Только для **периодического опроса** (не для `/start`): для каждой подходящей вакансии, чей `vacancy_id` **отсутствует** в хранилище FR-04, бот отправляет в сохранённый `chat_id` одно сообщение с кратким представлением вакансии, **добавляет** этот `vacancy_id` в конец очереди FR-04 (сохраняя уникальность: повторно добавлять тот же id не требуется) и выполняет **FR-04b**. Вакансии, чей id уже в FR-04, **автоматически** повторно не отправляются.

### 4.2. Команда `/start`: первичная выдача

**FR-07.** После сохранения `chat_id` выполняется получение и отбор подходящих вакансий по правилам подразделов **4.3–4.5**.

**FR-08.** Для `/start` порядок «свежести» задаётся полем **`published_at`** из данных hh.ru (убывание). Допускается опора на параметры сортировки `GET /vacancies` из OpenAPI при условии, что итоговый порядок для топ-10 совпадает с сортировкой по `published_at` убыв.

**FR-09.** Выдача по `/start` определяется **только** правилом **L-02** (до 10 самых свежих по hh.ru среди подходящих по запросу/фильтрам).

**FR-10.** Содержимое сообщения (на одну вакансию или с разбивкой по лимитам Telegram): название вакансии, работодатель, город и/или формат работы, публичная ссылка (поле `alternate_url` или эквивалент из выдачи API), величина зарплаты при наличии в данных.

**FR-11.** После отправки выдачи по `/start` каждый `vacancy_id` из этой выдачи, **если его ещё нет** в FR-04, **добавляется** в конец очереди FR-04; затем выполняется **FR-04b**. Так автоопрос (**L-03**) не дублирует эти id, пока они остаются в хранилище. Повторный `/start` снова выполняет **L-02** независимо от состава FR-04.

### 4.3. Фильтр профессиональной роли

**FR-12.** В запросах поиска передаётся `professional_role=96` (роль «Программист, разработчик»; справочник `GET https://api.hh.ru/professional_roles`). При изменении справочника со стороны hh.ru исполнитель сверяет актуальность значения по официальному ответу API.

**FR-12a.** Фильтр опыта: в **каждом** запросе `GET /vacancies` передаются **оба** значения параметра `experience`: **`noExperience`** и **`between1And3`** (идентификаторы из `GET https://api.hh.ru/dictionaries`, ключ `experience`) — в виде **двух** одноимённых query-параметров (`experience=noExperience&experience=between1And3` или эквивалент в HTTP-клиенте). В [OpenAPI hh.ru](https://api.hh.ru/openapi/redoc) для `experience` указано, что **можно указать несколько значений**; это соответствует отбору вакансий **без опыта** **или** с опытом **1–3 года** (логика **ИЛИ**), без отдельных запросов только ради опыта.

### 4.4. Фильтр география / формат работы

**FR-13.** Вакансия считается подходящей по географии/формату, если выполняется: локация соответствует **Пермь** (согласно данным вакансии и `PERM_AREA_ID` / справочнику областей) **ИЛИ** в данных вакансии указан **удалённый** формат работы (идентификатор расписания по OpenAPI/справочникам).

**FR-14.** Вакансии, для которых не выполняется FR-13, в выдачу и уведомления не включаются.

**FR-15.** Реализация поиска: выполняется **два** базовых запроса `GET /vacancies` (без учёта постраничного продолжения): **(1)** ветка **Пермь** — параметр области (`PERM_AREA_ID` / OpenAPI), **FR-12**, **FR-12a**; **(2)** ветка **удалённая работа** — параметр `schedule` = **`SCHEDULE_REMOTE_ID`** (**INT-05**), **FR-12**, **FR-12a**. В **каждом** запросе задаётся сортировка выдачи по дате публикации **по убыванию** (`order_by` = `publication_time` — идентификатор из справочника `vacancy_search_order` в OpenAPI / `GET /dictionaries`). Сравнение дат при сортировке и **FR-15b** — по полю **`published_at`** из элементов ответа, парсинг как дата-время с учётом часового пояса из строки API. Обе выборки **объединяются** без дубликатов по `vacancy_id`; затем **единая** сортировка по `published_at` по убыванию для применения **L-02** и **L-03**. Постраничный обход (**FR-15b**, **INT-06**, **L-03**, **NFR-05a**) выполняется **для каждого** из **двух** базовых запросов независимо. При необходимости уточнение по полной карточке `GET /vacancies/{id}` для проверки FR-13 и FR-18.

**FR-15b. Ранняя остановка пагинации (не ходить в «уже просмотренный хвост»):** для **каждой** ветки (Пермь / удалённая работа) страницы `page=0,1,…` запрашиваются, пока **не** сработает **любое** из условий остановки: (0) нарушен предел глубины выдачи **INT-06**; (1) API не возвращает данных или страница пуста; (2) достигнут лимит запросов **NFR-05a**; (3) на **текущей** странице **все** вакансии имеют `vacancy_id` уже в **FR-04**; (4) для этой ветки в state задан **`pagination_floor_*`** (**FR-04c**) **и** минимальный `published_at` среди вакансий **на текущей странице** **≤** этого порога **и** на странице **нет** ни одной вакансии с `vacancy_id ∉ FR-04`. **После** завершения обхода ветки в рамках цикла **L-03** обновить соответствующее поле **FR-04c**: записать **минимальный** `published_at` среди **всех** элементов, полученных с API по этой ветке в **этом** цикле; если ветка в цикле не вызывалась — поле **не** менять. Для **`/start`** применяются **те же** правила **FR-15b** по **каждой** ветке (с **чтением** **FR-04c**, без обновления порогов — **FR-04c**), **без** эвристики «остановить ветку, как только набралось 10 подходящих»: топ-10 в **L-02** определяется только после **объединения** двух веток и **глобальной** сортировки, иначе возможна ошибка (все 10 лучших с одной ветки, вторая не догружена).

### 4.5. Фильтр ключевых слов

**FR-16.** Набор ключевых слов задаётся только в коде конфигурации и/или переменных окружения; ввод списка через Telegram в рамках ТЗ не требуется.

**FR-17.** Фиксированный набор ключевых слов (регистр символов **не** учитывается): `c#`, `c++`, `go`, `golang`, `java`, `python`, `.net`. Совпадение **только как отдельный токен** в объединённом тексте **названия и описания** (после нормализации пробелов), **не** как вхождение внутри другого слова: например, `java` **не** считается найденным внутри `javascript`; `go` **не** считается найденным внутри `golang` (для `golang` используется отдельное ключевое слово из списка). Для `c#`, `c++`, `.net` используются шаблоны, исключающие совпадение «по части» другого идентификатора (экранирование спецсимволов и проверка границ токена в реализации). Допустима реализация через регулярные выражения с явными границами токена, эквивалентные по смыслу примерам выше.

**FR-18.** Вакансия подходит по ключевым словам, если хотя бы одно значение из FR-17 удовлетворяет правилам FR-17 в **названии** вакансии или в **тексте описания** (описание — из полной карточки при отсутствии в краткой выдаче).

**FR-19.** Фильтрация по FR-17–FR-18 выполняется на стороне приложения; параметр `text` в запросе поиска hh.ru не считается достаточным для замены FR-18.

## 5. Нефункциональные требования

**NFR-01.** Язык реализации: Python 3.11+.

**NFR-02.** HTTP-клиент: `httpx` (асинхронный режим согласован с обработчиками бота).

**NFR-03.** Библиотека Telegram: `python-telegram-bot` версии 20 или выше; для периодических задач — зависимость `python-telegram-bot[job-queue]` (APScheduler).

**NFR-04.** Развёртывание: контейнеризация Docker; `docker-compose.yml` с монтированием тома для файла состояния; секреты не включаются в образ и не коммитятся (`.env` в `.gitignore`).

**NFR-05.** При ответе API **429** выполняется ожидание (backoff) и протоколирование без вывода секретов.

**NFR-05a.** За один 5-минутный цикл **L-03** задаётся **верхний предел** числа **всех** HTTP-запросов к `api.hh.ru` в этом цикле — включая **GET /vacancies** (все страницы обеих веток) и **GET /vacancies/{id}** для полных карточек (конфигурация, например переменная окружения). Постраничный обход по **FR-15** и **FR-15b** продолжается, пока не сработает **ранняя остановка**, **INT-06** или предел **не достигнут**; при достижении предела цикл **корректно завершается** (допускается догрузка в следующем тике). Лимит согласуется с правилами и устойчивостью API hh.ru.

**NFR-06.** Логирование не содержит значений токенов и иных секретов.

**NFR-07.** После перезапуска контейнера состояние восстанавливается с тома; **периодический опрос** не отправляет вакансии, чьи `vacancy_id` уже содержатся в FR-04 (**L-03**). Повторная отправка тех же вакансий при очередном `/start` по **L-02** не считается нарушением NFR-07.

**NFR-08 (лимит FR-04).** В состоянии хранятся только **идентификаторы** вакансий (не полные описания). Максимум **1000** уникальных `vacancy_id` с политикой **FR-04b** (FIFO). Вытеснённые из FR-04 идентификаторы **могут снова** быть отправлены **автоопросом**, если вакансия снова попадёт в выборку и для пользователя это будет «новое» отсутствие id в текущем FR-04.

## 8. Критерии приёмки

1. Секрет бота только в `.env`; в логах секретов нет (NFR-06).
2. После каждого `/start` в чат уходит не более 10 подходящих вакансий — **самые свежие по `published_at` на hh.ru** среди подходящих (**L-02**); повторный `/start` заново запрашивает данные и может снова отправить те же вакансии. Хранилище FR-04 обновляется по FR-11 и **FR-04b** (лимит 1000).
3. Каждые 5 минут выполняется цикл опроса; каждая подходящая вакансия доставляется **автоматически** не более одного раза на `vacancy_id` (пока id хранится в множестве FR-04) (FR-05, FR-06). Постраничный обход соответствует **FR-15b** и обновлению **FR-04c** после цикла.
4. В выдаче отсутствуют вакансии, не удовлетворяющие FR-13–FR-14.
5. Все показанные вакансии удовлетворяют FR-12, **FR-12a** и FR-18.
6. Запуск на VPS по инструкции `README.md` с Docker и томом состояния успешен (NFR-04, NFR-07).
7. Один закреплённый `chat_id`; запрос `/start` из другого чата не переназначает получателя (FR-02).
8. Пагинация не нарушает **INT-06**; учёт запросов за цикл **L-03** соответствует **NFR-05a** (включая `GET /vacancies/{id}`).

## 9. Отладка и цикл согласования с ТЗ (исполнитель / ИИ-агент)

Ниже — обязательный порядок работ для исполнителя, в том числе при генерации кода по данному ТЗ (промпт = документ целиком или ссылка на него).

**DBG-01. Реализация по ТЗ.** Реализовать программу в соответствии с разделами 1–6, планом работ (раздел 7) и критериями приёмки (раздел 8).

**DBG-02. Автотесты.** Добавить и **запускать** тесты (например `pytest`), покрывающие по возможности: доменную логику (FR-12–FR-19, **FR-15b**, merge веток, сортировка по `published_at`), персистентность state (**FR-04**, **FR-04b**, **FR-04c**), граничные случаи **FR-17** (токены, `java`/`javascript`, `go`/`golang` и т.п.). Интеграционные тесты к живому `api.hh.ru` — по усмотрению (могут быть отключены по флагу), но **юнит-тесты с моками ответов API** предпочтительны для стабильности CI.

**DBG-03. Проверка фактической выдачи.** После прохождения тестов исполнитель **просматривает результаты**: вывод тестов, при необходимости — логи или контрольный прогон (dry-run / фикстуры), сверяя поведение с **L-02**, **L-03**, **FR-13–FR-19**, **FR-02**, **NFR-05a**, **INT-06** и пунктами раздела 8.

**DBG-04. Исправление при расхождениях.** Если фактическое поведение или результаты тестов **не совпадают** с ТЗ и критериями приёмки — **исправить код** (или тесты, только если ошибка в ожиданиях теста, а не в ТЗ), затем **повторить DBG-02–DBG-03** до согласованности. Зафиксировать в `README.md` только объективные ограничения среды (например отсутствие сети в CI), а не отступления от ТЗ без согласования.

**DBG-05. Завершение цикла.** Считать этап отладки закрытым, когда тесты стабильно проходят и проверка по **DBG-03** не выявляет противоречий разделам 4–5 и 8 (либо явно задокументированы внешние блокеры, не исправляемые в коде).
